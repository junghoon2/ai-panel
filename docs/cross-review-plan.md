# 교차 리뷰 기능 구현 계획

> 본 문서는 계획 전용이다. 실제 작업의 진행 상태는 `docs/progress.md`에서 관리한다 (SSOT 원칙).

## 목표
- 답변 완료 후 `/review <리뷰어> <대상>` 명령으로 한 도구가 다른 도구의 답변을 리뷰한다
- `/review all` 로 3개 도구가 서로의 답변을 교차 리뷰한다 (3패널 동시 스트리밍)
- 리뷰는 리뷰어의 기존 대화 세션을 이어서 수행해, 자기 답변과 비교하는 깊은 리뷰가 가능하다
- 리뷰 후에도 일반 후속 질문이 기존 맥락대로 동작한다

---

## 사전 확인 사항
- 기존 구조 재사용: 어댑터/세션 resume 은 변경 없음 (리뷰도 "프롬프트가 다른 질문 턴"일 뿐)
- CLI 인자 길이: 리뷰 프롬프트에 대상 답변 전문이 들어가지만, macOS ARG_MAX(수백 KB)와 모델 컨텍스트 대비 일반 답변 길이는 충분히 작음

## 아키텍처 결정

| 결정 | 선택 | 근거 |
|------|------|------|
| 명령 형식 | `/review <리뷰어> <대상>`, `/review all` | 사용자 선택. TUI 에서 슬래시 명령이 명확 |
| 결과 표시 | 리뷰어 패널에 표시 (새 턴처럼) | 사용자 선택. 기존 레이아웃 유지, 대상 패널의 원본 답변과 나란히 비교 가능 |
| 리뷰 맥락 | 리뷰어의 기존 세션 이어서 (resume) | 사용자 선택. 원래 질문·자기 답변을 기억한 채 비교 리뷰 |
| 리뷰 대상 데이터 | 마지막 **일반 질문 턴**의 답변만 (`lastAnswers`) | 리뷰 결과를 다시 리뷰 대상으로 삼으면 연쇄 혼란 → 리뷰 턴은 `lastAnswers` 를 갱신하지 않음 |
| orchestrator 확장 | `runTasks([{name, question}])` 로 일반화, 기존 `runQuestion` 은 래퍼로 유지 | `/review all` 은 도구마다 프롬프트가 달라 단일 question fan-out 으로는 불가 |
| 명령 오류 표시 | 헤더 아래 notice 줄 1개 추가 | 잘못된 도구명, 답변 없음 등의 피드백 위치 필요 |

리뷰 프롬프트(안) — 리뷰어 세션에 이어 보내므로 원래 질문은 "방금 질문" 으로 참조 가능하지만, 명확성을 위해 포함:

```
다음은 같은 질문에 대해 다른 AI 도구(<대상>)가 내놓은 답변이다.

원래 질문: <마지막 일반 질문>

--- <대상> 의 답변 ---
<답변 전문>
---

위 답변을 리뷰해줘. 정확성(틀린 내용), 누락된 중요 정보, 너의 답변과의 차이점을 중심으로 간결하게.
```

`/review all` 은 리뷰어마다 나머지 도구들의 답변을 위 형식으로 묶어 하나의 프롬프트로 보낸다.

데이터 흐름:

```
일반 질문 턴 done ──> lastAnswers[도구] = 답변 텍스트 저장
                      lastUserQuestion = 질문 저장

/review claude gemini
  ├─ 검증: 도구명 유효? 리뷰어 ≠ 대상? lastAnswers[gemini] 존재?
  ├─ 프롬프트 빌드 (lastUserQuestion + lastAnswers[gemini])
  └─ runTasks([{claude, 프롬프트}])  ← claude 세션 resume
        └─ claude 패널만 running → 리뷰 스트리밍 (lastAnswers 갱신 안 함)

/review all
  └─ runTasks([
       {claude, codex+gemini 답변 리뷰 프롬프트},
       {codex,  claude+gemini 답변 리뷰 프롬프트},
       {gemini, claude+codex 답변 리뷰 프롬프트},
     ])  ← 3패널 동시 스트리밍
```

## 작업 Phase

### Phase 1 — 기반 작업 (orchestrator 일반화 + 상태 저장)
**목표:** 리뷰 기능이 올라탈 토대를 기존 동작 회귀 없이 마련

- orchestrator 에 `runTasks(tasks: {name, question}[])` 추가, `runQuestion` 은 래퍼로 전환
- App 에 마지막 일반 질문(`lastUserQuestion`)과 도구별 답변(`lastAnswers`) 저장 (done 시점, 일반 턴만)
- 헤더 아래 notice 줄 추가 (명령 피드백용, 평소엔 숨김)
- 검증: 빌드 + tmux 로 기존 질문/후속 질문/종료 플로우 회귀 확인

### Phase 2 — `/review <리뷰어> <대상>` 단일 리뷰
**목표:** claude 가 gemini 답변을 리뷰하는 핵심 시나리오 동작

- submit 에서 `/review` 명령 파싱 (도구명 검증, 리뷰어=대상 차단, 대상 답변 존재 확인 → 실패 시 notice)
- 리뷰 프롬프트 빌더 작성 (`src/review.ts`)
- 리뷰어 패널만 running 으로 리셋하고 `runTasks` 실행 (세션 resume, `lastAnswers` 갱신 안 함)
- 리뷰 중 헤더에 `리뷰: <리뷰어> ← <대상>` 표시
- 검증: tmux 로 질문 → `/review claude gemini` → claude 패널에 리뷰 표시·gemini 패널 원본 유지 → 후속 일반 질문 맥락 정상

### Phase 3 — `/review all` + 마감
**목표:** 3개 도구 교차 리뷰와 오류 케이스 마무리

- `/review all` 파싱 → 도구별 교차 리뷰 프롬프트로 `runTasks` 3건 동시 실행
- 답변이 없는 도구는 리뷰 대상에서 제외 (notice 로 안내), 미설치 도구 조합 처리
- 오류 케이스 정리: `/review` 인자 부족, 알 수 없는 도구, 답변 전 실행
- README 사용법 갱신
- 검증: tmux E2E — 질문 → `/review all` 3패널 교차 리뷰 → 후속 질문 정상, 오류 명령 notice 확인

## 주요 리스크 및 대응

| 리스크 | 영향 | 대응 |
|-------|------|------|
| 리뷰 결과가 다음 리뷰의 대상이 되는 연쇄 혼란 | 리뷰의 의미 불명확 | 리뷰 턴은 `lastAnswers` 갱신 제외 — 리뷰 대상은 항상 마지막 일반 질문의 답변 |
| 리뷰 턴이 세션 히스토리에 쌓여 후속 질문 맥락이 흐려짐 | 후속 답변 품질 저하 | 허용 (사용자 선택: 세션 이어서). 문제 시 "새 세션 리뷰" 옵션을 추후 확장으로 |
| 대상 답변이 매우 길 때 프롬프트 비대화 | 토큰 낭비, 응답 지연 | v1 은 전문 전달. 추후 확장에서 길이 제한 옵션 검토 |
| gemini 리뷰어의 `--resume latest` 가 리뷰 턴도 최근 세션으로 이어감 | 동작 자체는 동일 (이어가기) | 기존 메커니즘 그대로 — 추가 처리 불필요함을 Phase 2 검증에서 확인 |
| 명령 파싱 분기 증가로 submit 비대화 | 유지보수성 저하 | 명령 파싱을 별도 함수로 분리 (`parseCommand`) |

## 완료 기준
- 질문 후 `/review claude gemini` 실행 시 claude 패널에 gemini 답변 리뷰가 스트리밍되고, gemini 패널의 원본 답변은 유지된다
- `/review all` 실행 시 3개 패널에 교차 리뷰가 동시 스트리밍된다
- 리뷰 후 일반 후속 질문이 기존 대화 맥락을 유지한다
- 잘못된 리뷰 명령(없는 도구, 답변 전 실행 등)은 notice 로 안내되고 앱은 정상 동작한다

## 추후 확장 (본 계획 범위 밖)
- 종합 심판 모드 (3개 답변 + 교차 리뷰를 받아 최종 판정하는 4번째 호출)
- 전체 폭 리뷰 뷰 (리뷰 동안 레이아웃 전환)
- 새 세션 독립 평가 옵션 (`/review --fresh`)
- 리뷰 프롬프트 커스터마이즈, 대상 답변 길이 제한
